# Accessibility Audit Report

**Site:** nmotlagh.github.io
**Date:** 2026-03-28
**Standard:** WCAG 2.1 AA + APCA where noted
**Scope:** All Astro source in `src/`, theme CSS, public assets (resume PDF)

---

## Critical

### C1. `--color-text-soft` fails contrast in both themes

`#94a3b8` on light backgrounds (~2.5-2.8:1) and `#64748b` on dark backgrounds (~2.1-3.3:1) both fail WCAG AA 4.5:1 for normal text. This token is used pervasively:

- `.section__subtitle` — section subtitles
- `.timeline-item__date` / `.timeline-item__location` — experience dates and locations
- `.card__meta` — card metadata
- `.publication-card__authors` — author names on every publication card
- `.highlight-card__eyebrow` — venue/year labels
- `.news-strip__items time` — news timestamps
- `.site-footer` — copyright text
- `.blog-post__header time` — blog post dates
- Inline `style="color: var(--color-text-soft)"` on artifact "Supports:" text

**Files:** `src/styles/theme.css` (lines 7, 73), all components using this token

### C2. Dark mode button text fails contrast

White (`#fff`) on `--color-accent` (`#60a5fa`) yields ~3.0:1. The hover state (`#93c5fd`) is worse at ~1.8:1. Every primary CTA button in dark mode fails AA.

**File:** `src/styles/theme.css:568`

### C3. Multiple `<h1>` on every page

The sidebar renders `<h1 class="sidebar__name">` (BaseLayout.astro:164) while every page also has its own `<h1>`. This produces two `<h1>` elements on every page that uses the sidebar layout, including index, 404, blog posts, and publication detail pages.

**File:** `src/layouts/BaseLayout.astro:164`

### C4. Resume PDF is not tagged

`pdfinfo` reports `Tagged: no`, `Title: (empty)`, `Author: (empty)`. The PDF has no document title, no structural tags, no alt text on any content, and no reading order metadata. Screen readers will attempt raw text extraction, which often garbles multi-column or styled LaTeX output. Generated by `pdfTeX-1.40.26` with `hyperref` (links work, but no structure).

**File:** `public/resume.pdf`

## Major

### M1. Heading skip: h2 to h4 in Skills section

Skills section uses `<h2>` for "Skills" then `<h4>` for each category, skipping `<h3>`.

**File:** `src/pages/index.astro:171,175`

### M2. Callout `<h4>` appears before any h2/h3

The "Currently" callout renders a hardcoded `<h4>` (Callout.astro:12) that appears in the document before the first `<h2>`, creating a severe heading-order violation on the homepage.

**Files:** `src/components/Callout.astro:12`, `src/pages/index.astro:125`

### M3. No `aria-live` announcement for theme changes

When the theme toggle is clicked, `aria-pressed` updates but no live region announces the result. Screen reader users get no confirmation of what changed.

**File:** `src/components/ThemeToggle.astro`

### M4. Navigation links not in a list

The `<nav>` in the sidebar contains bare `<a>` elements, not wrapped in `<ul>/<li>`. Screen readers cannot announce the number of navigation items.

**File:** `src/layouts/BaseLayout.astro:192-196`

### M5. No mobile navigation

The sidebar `<nav>` is `display: none` on mobile (theme.css:248) with no hamburger menu or alternative. Keyboard and touch users on narrow viewports cannot access in-page navigation links.

**File:** `src/styles/theme.css:248`

### M6. Contact/social links lack `<nav>` and list markup

Five contact links (Email, GitHub, Scholar, LinkedIn, Resume) are in a plain `<div>` with no `<nav aria-label="...">` or `<ul>` wrapper.

**File:** `src/layouts/BaseLayout.astro:169-190`

### M7. Multiple unlabeled `<aside>` landmarks

The sidebar `<aside>` and each `Callout` `<aside>` lack `aria-label`, making them indistinguishable to screen reader landmark navigation. (Hero's `<aside aria-label="Featured portrait">` is correctly labeled.)

**Files:** `src/layouts/BaseLayout.astro:153`, `src/components/Callout.astro:11`

### M8. `<footer>` nested inside `<main>`

The site footer is a child of `<main>` in both layout variants. Screen readers may not announce it as a page-level footer landmark.

**File:** `src/layouts/BaseLayout.astro:207-210, 218-221`

### M9. No page-level `<header>` (banner) landmark

The sidebar contains site identity and navigation but is wrapped in `<aside>`, not `<header>`. Screen reader users navigating by landmarks will not find a banner region.

**File:** `src/layouts/BaseLayout.astro`

### M10. Sidebar SVGs missing `aria-hidden="true"`

Each contact link's inline `<svg>` lacks `aria-hidden="true"`. Since visible text labels follow each icon, the SVGs are decorative and may cause screen readers to announce raw SVG content.

**File:** `src/layouts/BaseLayout.astro:171,175,179,183,187`

### M11. Theme toggle missing consistent `:focus-visible` style

The `.theme-toggle` class has no `:focus-visible` rule and is not included in the grouped focus-visible selector at theme.css:229. It falls back to the browser default outline, inconsistent with the site's `2px solid var(--color-accent)` pattern.

**File:** `src/styles/theme.css` (missing rule for `.theme-toggle:focus-visible`)

### M12. Theme toggle touch target is 38px

At 38x38px, the toggle falls short of the 44px mobile target (WCAG 2.5.8).

**File:** `src/styles/theme.css:678-679`

## Minor

### m1. Badge links lack contextual `aria-label`

Links labeled "Paper", "arXiv", "Code" inside publication cards are ambiguous when listed out of context (e.g., screen reader link list). Adding `aria-label="Paper for {title}"` would help.

**Files:** `src/components/PublicationCard.astro:44-56`, `src/components/HighlightsRow.astro:40-49`

### m2. Duplicate "GitHub" link text across artifact cards

Multiple artifact cards each have a link reading just "GitHub", indistinguishable in a link list.

**Files:** `src/pages/artifacts/index.astro:36`, `src/pages/publications/[slug].astro:100`

### m3. Tags, badges, and skills use `<div>/<span>` instead of `<ul>/<li>`

- `.card__tags` in PublicationCard (spans, not list items)
- TechBadges component (spans in a div)
- Skill items on homepage (spans in divs)

These are semantically lists and would benefit from proper list markup.

**Files:** `src/components/PublicationCard.astro:32-38`, `src/components/TechBadges.astro:26-33`, `src/pages/index.astro:176-225`

### m4. `<ol>` with `list-style: none` missing `role="list"`

Safari strips list semantics when `list-style: none` is applied. The Hero component's `<ul>` correctly adds `role="list"` to compensate, but `<ol>` elements in NewsStrip, news archive, and blog index do not.

**Files:** `src/components/NewsStrip.astro:18`, `src/pages/news/index.astro:22`, `src/pages/blog/index.astro:25`

### m5. Highlight card titles are not headings

Titles in HighlightsRow are `<a>` elements, not `<h3>`. Screen reader users navigating by headings will skip these items.

**File:** `src/components/HighlightsRow.astro:35`

### m6. Static `aria-label` on theme toggle

Label is always "Toggle theme" regardless of state. While `aria-pressed` communicates on/off, the meaning of pressed=dark is not self-evident. A dynamic label like "Switch to dark theme" / "Switch to light theme" would be clearer.

**File:** `src/components/ThemeToggle.astro:1`

### m7. Nine instances of `transition: all`

Violates the project's own CLAUDE.md guideline. Can cause layout thrashing on low-powered devices. Found in:

| Selector | File:Line |
|---|---|
| `.sidebar__link` | `theme.css:222` |
| `.sidebar__nav-link` | `theme.css:255` |
| `.tech-badge` | `theme.css:288` |
| `.timeline-item__content` | `theme.css:450` |
| `.card` | `theme.css:489` |
| `.button` | `theme.css:572` |
| `.theme-toggle` | `theme.css:683` |
| `.highlight-card` | `theme.css:745` |
| `.highlight-card` (scoped) | `HighlightsRow.astro:114` |

### m8. `.button--small` has no `min-height`

May result in sub-44px touch targets on mobile, unlike `.button` which correctly sets `min-height: 44px`.

**File:** `src/styles/theme.css:593`

### m9. No `:visited` link styles

All links appear identical whether visited or not. Not a WCAG failure but a usability concern.

### m10. Smallest text is 12px

`.tag-list li` uses `0.75rem` (12px). Legibility concern for low-vision users, though not a strict WCAG failure.
